# ChatALL Web 性能优化方案

## 1. 虚拟滚动实现方案

### 1.1 核心组件

#### VirtualChatMessages 组件
**位置**: `src/components/Messages/VirtualChatMessages.vue`

**核心特性**:
- **动态高度计算**: 使用 Map 存储每个消息项的实际高度
- **视口缓冲**: 前后各保留 2 个缓冲项，提升滚动流畅度
- **节流滚动**: 使用 `requestAnimationFrame` 节流，避免过度渲染
- **无限滚动**: 自动触发下一页加载

**关键配置**:
```javascript
const estimatedHeight = 250      // 默认估算高度
const viewportBuffer = 2        // 视口缓冲数量
const scrollThreshold = 500     // 加载阈值(px)
```

#### VirtualScrollMessages 组件
**位置**: `src/components/Messages/VirtualScrollMessages.vue`

**适用场景**: 简单列表场景，提供基础虚拟滚动能力

### 1.2 性能优化策略

| 优化项 | 实现方式 | 预期效果 |
|--------|----------|----------|
| DOM 节点控制 | 仅渲染可见区域 | 节点数从 N 降至 10-20 |
| 滚动事件节流 | throttle + rAF | 避免高频重绘 |
| 高度缓存 | Map 存储已测高度 | 避免重复计算 |
| 懒加载 | 分页加载消息 | 减少初始加载量 |

### 1.3 无限滚动机制

```
滚动触发 → 距离底部 < 500px → 触发 load-more → 加载下一页(50条) → 更新列表
```

## 2. 缓存策略设计

### 2.1 多级缓存架构

```
┌─────────────────────────────────────────────────────────┐
│                    应用层缓存 (LRU)                      │
│  ┌──────────┐  ┌──────────┐  ┌─────────────┐          │
│  │ chatCache│  │botConfig │  │apiResponse  │          │
│  │  (50)    │  │Cache(100)│  │Cache(200)   │          │
│  └──────────┘  └──────────┘  └─────────────┘          │
├─────────────────────────────────────────────────────────┤
│                  浏览器缓存 (IndexedDB)                   │
│              ┌─────────────────────┐                    │
│              │    Dexie DB         │                    │
│              │ chats/messages/threads│                  │
│              └─────────────────────┘                    │
├─────────────────────────────────────────────────────────┤
│                    HTTP 缓存                            │
│              Cache-Control / ETag                       │
└─────────────────────────────────────────────────────────┘
```

### 2.2 LRU 缓存实现

**位置**: `src/utils/cache.js`

**缓存配置**:

| 缓存类型 | 最大容量 | TTL | 清理周期 |
|----------|----------|-----|----------|
| 聊天记录 | 50 条 | 5分钟 | 5分钟 |
| Bot 配置 | 100 条 | 30分钟 | 5分钟 |
| API 响应 | 200 条 | 可配置 | 5分钟 |

**核心方法**:
```javascript
get(key)      // 获取缓存，自动更新访问顺序
set(key, value) // 设置缓存，超出容量时淘汰最久未使用
has(key)      // 检查缓存是否存在
delete(key)   // 删除缓存项
```

### 2.3 缓存同步策略

- **写入时**: 同时更新 IndexedDB 和内存缓存
- **读取时**: 优先读取内存缓存，失效时从 DB 加载
- **删除时**: 同时清理两级缓存

## 3. 状态管理优化

### 3.1 批量更新机制

**位置**: `src/utils/debounce.js`

**防抖更新**:
```javascript
const debouncedSave = debounce(async (queue) => {
  const updates = queue.map(item => 
    db.messages.update(item.index, item.message)
  )
  await Promise.all(updates)
}, 100)
```

**批量操作**:
- 消息更新累积到队列
- 100ms 后批量写入数据库
- 减少 IO 操作次数

### 3.2 响应式优化

**计算属性缓存**:
```javascript
const groupedMessages = computed(() => {
  // 仅在 messages 变化时重新计算
  const grouped = []
  // ... 分组逻辑
  return grouped
})
```

**避免嵌套响应式**:
- 使用 Map 替代 Array 存储高度
- 减少响应式依赖链

### 3.3 数据结构优化

| 原结构 | 优化后 | 优势 |
|--------|--------|------|
| Array 查找 | Map 查找 | O(1) vs O(n) |
| 线性遍历 | 索引映射 | 快速定位 |
| 嵌套对象 | 扁平化 | 减少响应式开销 |

## 4. 性能指标目标

### 4.1 核心指标

| 指标 | 目标值 | 当前值 | 提升幅度 |
|------|--------|--------|----------|
| DOM 节点数 | < 500 | ~5000+ | >90% |
| 首屏渲染时间 | < 100ms | ~500ms | >80% |
| 滚动 FPS | > 55 | ~30 | >80% |
| 内存占用 | < 200MB | ~500MB | >60% |
| 消息加载时间 | < 50ms | ~200ms | >75% |

### 4.2 监控工具

**位置**: `src/utils/performance.js`

**监控指标**:
- `renderTime`: 渲染耗时
- `scrollTime`: 滚动处理耗时
- `memoryUsage`: 内存占用(MB)
- `domNodes`: DOM 节点数
- `fps`: 帧率

**监控方法**:
```javascript
logPerformanceReport()  // 输出性能报告
shouldOptimize()        // 判断是否需要优化
calculateFPS()          // 计算实时帧率
```

### 4.3 优化触发条件

当满足以下任一条件时自动触发深度优化：
- FPS < 30
- DOM 节点数 > 5000
- 内存占用 > 300MB

## 6. 代码规范

### 6.1 性能相关规范

1. **避免频繁响应式更新**
   - 使用 `shallowRef` 替代 `ref` 存储大量数据
   - 批量更新使用队列机制

2. **合理使用计算属性**
   - 缓存计算结果
   - 避免在计算属性中执行复杂逻辑

3. **事件处理优化**
   - 滚动/输入事件必须节流/防抖
   - 使用 `requestAnimationFrame` 进行视觉更新

4. **内存管理**
   - 组件卸载时清理引用
   - 定期清理过期缓存

### 6.2 性能审查清单

- [ ] 列表渲染是否使用虚拟滚动
- [ ] 事件处理是否节流/防抖
- [ ] 大数据量是否分页加载
- [ ] 重复计算是否缓存
- [ ] 组件是否及时清理
